For a **hedge-fund-style Binance futures bot**, don’t think in “indicators.” Think in **signal categories / models**.

A real systematic desk usually has these top-level blocks:

```txt
Alpha model
Risk model
Transaction cost model
Portfolio construction model
Execution model
Post-trade analytics
```

That structure is common in quant trading literature: alpha predicts opportunity, risk limits exposure, transaction-cost models estimate fees/slippage/impact, portfolio construction chooses size/allocation, and execution decides how to trade efficiently. ([cacm.acm.org][1])

For our Binance futures bot, I would configure **25 signal categories**.

# Hedge-fund bot signal taxonomy

## 1. Price action / structure signals

These are basic but still useful.

```txt
higher highs / lower lows
break of structure
range break
range reclaim
failed breakout

```

Use: directional bias, invalidation, stop placement.

## 25. Model-health / strategy-decay signals

This is what most retail bots never build.


Model risk is real: financial models can fail due to wrong assumptions, data errors, programming errors, or changing conditions, so production systems need monitoring and governance. ([Investopedia][4])

Yes. For your **Binance futures bot**, scoring should not be one magic number like:

```txt
score > 70 = trade
```

That is too dangerous.

A pro bot should use **layered scoring**:

```txt
1. Setup Score
2. Regime Score
3. Execution Score
4. Risk Score
5. Net Edge Score
6. Safety Score
```

Then trade only if **all gates pass**.

# Final scoring model

Use a 0–100 score for each layer.

```txt
Final Trade Score =
  30% Setup Score
+ 20% Regime Score
+ 15% Execution Score
+ 15% Risk Score
+ 10% Net Edge Score
+ 10% Position Management Score
```

But safety is different:

```txt
Safety Score is not weighted.
If safety fails, no trade.
```

Meaning:

```txt
even if final score = 95
but stop placement risk is bad
NO TRADE
```

# 1. Setup Score

This measures whether the actual trading opportunity is strong.

For our main strategy, **liquidation exhaustion fade**:

```txt
Liquidation spike strength        0–20
Volume climax                     0–15
Price stretched from VWAP         0–15
Failed continuation               0–20
Order book refill / absorption    0–15
OI flush / leverage unwind        0–15
```

Total: `100`

Example:

```txt
liquidation spike strong       18/20
volume climax                  13/15
VWAP stretch                   12/15
failed continuation            17/20
book refill                    10/15
OI flush                       11/15

Setup Score = 81
```

Good.

---

# 2. Regime Score

This decides whether the market condition allows the strategy.

For exhaustion fade:

```txt
Panic/cascade regime present      0–25
Not clean trend continuation      0–20
Volatility not insane             0–15
BTC macro not breaking hard       0–15
Funding/sentiment supportive      0–10
Session liquidity acceptable      0–15
```

Total: `100`

Important:

```txt
If market is in strong clean trend,
do not fade just because price is stretched.
```

So for fade strategy:

```txt
trend continuation regime = bad
panic exhaustion regime = good
```

---

# 3. Execution Score

This is where most retail bots fail.

```txt
Spread normal                     0–20
Order book depth enough           0–20
Expected slippage low             0–20
Fill probability good             0–15
Latency/API healthy               0–15
No rate-limit / stream issue      0–10
```

Total: `100`

Hard rule:

```txt
Execution Score < 70 = no trade
```

Even if signal looks amazing.

---

# 4. Risk Score

This checks whether the trade is safe for the account.

```txt
Liquidation distance safe         0–20
Stop distance logical             0–20
Risk within $3–$5                 0–20
Daily loss budget available       0–15
No recent loss streak             0–10
Position exposure acceptable      0–15
```

Total: `100`

Hard rule:

```txt
Risk Score < 80 = no trade
```

For small capital, risk must be stricter than signal.

---

# 5. Net Edge Score

This asks:

```txt
after fees, funding, spread, slippage, is this still worth taking?
```

Score:

```txt
Expected R multiple               0–30
Expected fees acceptable          0–15
Funding cost acceptable           0–10
Slippage vs target acceptable     0–20
Historical win rate in same setup 0–15
Expected value positive           0–10
```

Total: `100`

Hard rule:

```txt
Net Edge Score < 65 = no trade
```

---

# 6. Position Management Score

Before entry, the bot should know how it will manage the trade.

```txt
Clear invalidation                0–25
Clear partial exit level           0–20
Clear breakeven trigger            0–15
Clear time-stop                    0–15
Clear full-exit rule               0–15
Reward/risk enough                 0–10
```

Total: `100`

Hard rule:

```txt
No clear exit plan = no trade
```

---

# 7. Safety Score

This is pass/fail.

No trade if any of these are true:

```txt
Binance user stream stale
WebSocket disconnected
REST reconciliation failed
existing position mismatch
open orphan orders
stop order cannot be placed
symbol precision not loaded
daily loss limit hit
weekly loss limit hit
2 consecutive losses hit
funding spike window blocked
news/event block active
spread abnormal
liquidation distance too close
```

Safety is not a score. It is a **circuit breaker**.

# Size by score

Do not use same size for every trade.

For $400 account:

```txt
Score 80–84:
  risk $2–$3

Score 85–89:
  risk $3–$4

Score 90+:
  risk $4–$5

After loss streak:
  cut risk by 50%

After daily profit target hit:
  stop or only A+ setup
```

Never increase risk just because you “feel” the setup is good.
